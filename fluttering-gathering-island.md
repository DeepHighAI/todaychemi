## 3. 핵심 피처

### 3.1 무료

- **내 사주 카드**: 본명식·일주·십신·오행 무제한 조회
- **합피드 슬롯 7명**: 별명·생년월일(+시)·성별·사이태그. *실명 수집 금지*
- **오늘합**: 매일 1장 무료 카드 (오늘의 운·주의점·행운 시간·키워드 3종)
- **가입 진입 토큰**: 가입 직후 *1회 무료 합보기 + D+1 1회 무료 다시합* 자동 지급
- **연속 출석 보상**: 7일 → 월간 흐름 미리보기, 14일 → 무료 리포트 1건

### 3.2 [핵심] 합보기 (8p)

본 제품의 단일 핵심 단위. 1회 호출당:

- **입력**: 본인 ChartCore + 인연 ChartCore + 6모드 응답 5-7문항
- **출력**: 양쪽 사주 만세력 풀 해석 + 호환성 / 강점 / 주의점 / 일단이거해봐 3 + 명리 고전 인용 0-2건(RAG)
- **6모드 (합플레이)**: 일합 / 친구합 / 돈합 / 첫합 / **썸합** / **오래합** (ADR-025). 모드 selector 화면 문구 의무 노출:
  - 일합 = "같이 일해도 잘 맞을까?"
  - 친구합 = "편하게 오래 갈 사이일까?"
  - 돈합 = "같이 돈 벌어도 괜찮을까?"
  - 첫합 = "한 번 더 만나도 될까?"
  - 썸합 = "이 설렘, 이어질까?"
  - 오래합 = "오래 함께해도 괜찮을까?"
- **LLM**: GPT-5, in 3,500 + out 800 (§7.1 + tech_stack.md §3.1과 일관)
- **PII**: chart_core + question_slot만 LLM 송신, 별명·birth_date 원본 절대 미전달

> "모르는사람(일회성 인연)" 모드는 신뢰성 한계로 제거하고, *새 인연을 대하는 본인 자기 성찰 카드*로 §3.4 진단 시리즈로 이동(ADR-025). "연애" 단일 모드는 *썸*과 *오래합*로 분리하여 초기 설렘/장기 합보기 포인트 차이를 명확히 함.

#### 3.2.1 호환성 산정식 (0-100 정수, 결정형)

| 가중치 | 명리 요소 | 산정 방법 |
|---|---|---|
| 40% | 합·형·충·해 | 양 일주 간 천간합·지지합·삼합·반합(+) / 형·충·파·해(−) 매핑 |
| 30% | 십신 상호작용 | 본인 일간 기준 인연의 십신 균형도 |
| 30% | 오행 균형 | 양 사주 오행 5지표 보완·과다 비율 |

LLM은 점수 산출에 개입하지 않고, 산정 결과를 *해석 본문 작성의 입력값*으로만 받는다.

> **개발 명세 분리 (ADR-035)**: 위 가중치 외 개발 구현에 필요한 *상세 점수 명세*는 별도 문서 **`compatibility_scoring_spec.md`** 로 분리한다. 해당 문서가 다루는 항목:
> - 천간합·지지합·삼합·반합·형·충·파·해 점수표
> - 합과 충 동시 발생 시 우선순위·중첩 처리
> - 오행 보완(부족 보완 +N) / 과다 중첩(중복 −N) 계산식
> - 시간 미상 시 시나리오 추정 ± 신뢰 구간 산정 방식
> - 6모드별 ±10p 가중치 재분배 매핑 테이블
> - 점수 클램프(0 미만 → 0, 100 초과 → 100) 처리
> - 결정성 보장: 동일 입력 → 동일 점수 반환 검증 절차
> 
> Phase 0 G1 단계에서 명리 specialist가 채워 검수, prompt_version과 별개의 `scoring_version` 필드로 관리(카나리 배포·롤백 가능).

#### 3.2.2 합카드 시각 컴포넌트 (Phase 1 잠금 / 7·8은 추가)

1. **헤더**: 본인 일주 chip ⊕ 인연 일주 chip (오행 컬러 토큰)
2. **합게이지**: 0-100 원형 + 4구간 라벨 (0-39 약함 / 40-59 보통 / 60-79 좋음 / 80-100 매우 좋음)
3. **오행맵**: 양쪽 사주 오행 5지표 중첩 표시
4. **본문 영역별 카피 (ADR-034 수정)** — *첫 viewport 요약*과 *자세히 보기*를 분리하여 짧고 명료함과 유료 결과의 납득감을 동시 확보:

| 영역 | 글자 수 | 구성 |
|---|---|---|
| **첫 화면 요약 (viewport 즉시)** | **150자 이내** | 결론 1줄 + 강점 1줄 + 주의점 1줄 + 일단이거해봐 3개 |
| **자세히 보기 (펼침)** | **400~600자** | 강점·주의점·일단이거해봐 각 단락 상세화 + 모드별 본문 변형 |
| **근거 보기 1단** | 표 + 짧은 해설 | 합·형·충·해 매핑·십신·오행 수치 + 직전 대비 변화 폭 (ADR-033) |
| **근거 보기 2단** | 고전 원문 + 번역 | 0-2건 (ADR-023) |
| **다시합 변화 설명** | **150~250자** | 직전 해석 대비 무엇이 바뀌었는가 + 변화 요인 명리 키워드 |

ChatGPT 사주 GPT(운세박사 300만 사용자)와 풀이 깊이 경쟁 회피하되, *유료 핵심 결과*는 자세히 보기 단계에서 충분한 납득감 제공. 첫 화면은 짧고 행동 위주, 자세히 보기는 깊이 (LLM 출력 동일 호출 안에서 길이 가변 처리)
5. **펼침 영역 "근거 보기 (명리)"**: 합·형·충·해 매핑, 십신 상호작용, 오행 수치 + **고전 원문 + 현대어 번역 병기 0-2건(RAG)**
6. **푸터**: 다시합 D+1/7/30 알림 예약 안내 + 합메모 입력칸 + 공유 CTA
7. **명리 용어 ⓘ 툴팁 (Phase 1 의무)**: 일주·십신·합·형·충·해 등 첫 등장 시 자동 노출, 2-3문장 + 고전 1줄 인용
8. **영역별 호환성 미니 레이더 (Phase 1.5)**: 일·사랑·돈·정서 4축 — 6모드와 직교 보강. 펼침형(기본 접힘). 별도 가격 없음

> 결정형 산정식(2·3·5) + LLM 본문(4) + 신뢰 강화 컴포넌트(7·8) 조합으로 LLM 신뢰도와 만세력 깊이를 동시에 시각화.

#### 3.2.3 합카드 viewport 우선순위 (ADR-026)

첫 진입 화면에는 **[1] 헤더·[2] 합게이지·[4] 본문 3섹션·[6] 푸터**만 노출하여 사용자가 핵심을 즉시 인지하게 한다. **[3] 오행맵·[5] 근거 보기·[7] 툴팁·[8] 미니 레이더는 스크롤·펼침·툴팁**으로 분리하여 정보 과부하를 방지한다.

| 컴포넌트 | viewport 처리 |
|---|---|
| [1] 헤더, [2] 합게이지, [4] 본문 3섹션(결론/강점/주의점/일단이거해봐) | **첫 진입 화면 즉시 노출 — 핵심 인지** |
| [3] 오행맵 | 스크롤 또는 "자세히 보기" 펼침 |
| [5] 근거 보기 | 펼침형 (기본 접힘). **1단(명리 산정 결과) → 2단(고전 원문 + 현대어 번역)** 으로 분리 — 일반 사용자 진입 장벽 완화 (ADR-023 수정) |
| [6] 푸터 (다시합 D+1 알림 / 합메모 / 공유 CTA) | **2차 액션 구역으로 분리** — 본문 스크롤 후 노출. 첫 인지 화면 노이즈 제거 |
| [7] 명리 용어 ⓘ 툴팁 + **바텀시트 + "쉽게 보기" 토글** | 첫 등장 자동 노출, 재호출 ⓘ 탭 + 용어 카드 바텀시트 + 본문 전체를 평이 표현으로 전환하는 "쉽게 보기" 토글 (ADR-023 강화) |
| [8] 영역별 미니 레이더 | 펼침형 (기본 접힘, Phase 1.5) |

> ADR-016(1~6 컴포넌트 잠금) 위계는 유지. ADR-026은 *순서·노출 시점*만 정의. *첫 진입 화면 = 결론·호환성·강점·주의점·일단이거해봐만 노출*하여 정보 과부하 완화. CTA(다시합/공유/알림 예약)는 결과 본문을 본 후 자연스럽게 만나도록 2차 액션화.

### 3.3 [핵심 확장] 관계 진화 타임라인 (4p 다시합)

**다시합 명리 근거**: 양쪽 본명식은 평생 불변. *현재 시점*의 일진(D+1)·주운(D+7)·월운/절기(D+30)이 양쪽 사주에 시간 변수로 투영되며 합·형·충·해와 십신·운기 점수가 변한다. 합카드 "근거 보기"에 *현재 일진/주운/월운*이 양쪽 일주에 미치는 영향을 명시.

- **타임라인 뷰**: 인연 디테일에서 등록/해석/다시합/합메모 시간순 표시
- **합메모**: 인연당 무제한 텍스트(80자 이내)
- **다시합 알림 D+1 기본 ON / D+7 · D+30 옵트인 (ADR-012 수정)**: D+1 푸시만 인연당 자동 1회 기본 추천. D+7/D+30은 인연 디테일 또는 합카드의 *"합켜두기"* 토글을 사용자가 켤 때만 활성화 — 푸시 피로(인연 7명 × 3회 = 최대 21회/월) 직접 대응
- **다시합 = 4p** (정가 50%, 동일 인연·동일 모드)
- **합흐름 그래프**: 시점별 점수 라인 차트
- **진단 스냅샷 영구 저장**: immutable 스냅샷(prompt_version·chart_hash·결과 본문). 탈퇴 시 익명화 후 통계만 보존
- **재방문 동인 다중화 (Phase 1.5, ADR-033)** — D+1 푸시 단일 의존 회피, *흐름 변화 가시성*을 다중 트리거로 표면화:
  1. **본인 운기 변화 → 인연 영향 푸시**: 본인 일진(D+1)/주운(D+7)/월운(D+30) 단위로 일주 합·형·충·해 변화가 큰 날, 등록 인연 중 *가장 영향 큰 1명* 자동 하이라이트 푸시. 카피 예: *"오늘 본인 일진이 ○○님과 강하게 연결돼요"*
  2. **그리드 자동 정렬 — 흐름이 가장 많이 바뀐 사람**: [합피드] 그리드 상단에 *합점수 변화 폭 ≥ ±10p* 인연 1명 자동 노출. "흐름 변화 큼" 배지. 클릭 시 4p 다시합 진입
  3. **변화 인지 임계값 시각화**: 합카드 [5] 근거 보기 1단에 *직전 해석 대비 변화 폭(±점수 + 변화 요인 명리 키워드)* 명시. 미묘한 변화도 사용자가 인지 가능

#### 3.3.1 다시합 결정성 명세 (ADR-036 신설)

같은 날짜에 같은 입력이면 같은 결과를 반환해야 한다 — 결정성 보장이 사용자 신뢰의 기반. 개발 구현 명세:

```
change_score = current_compatibility_score - previous_snapshot_score
change_reason = top 1~3 changed_factors (합·충·십신·오행 중)
trigger_threshold = abs(change_score) >= 10  // 그리드 자동 정렬·푸시 트리거
```

| 변화 폭 임계값 | **±10p** 잠정. Phase 1.5 출시 후 8주 실측으로 ±5p~±15p 범위 재조정 — [TBD-6] |
| 같은 날 재호출 결정성 | **동일 (chart_hash, scoring_version, prompt_version, 일진 date) → 캐시 히트 → 동일 점수**. 동일 날 두 번째 호출 시 LLM 본문은 캐시(§7.2 30일 TTL), 점수는 100% 결정적 |
| previous_snapshot 버전 기준 | **scoring_version + prompt_version 페어**가 같은 가장 최근 immutable 스냅샷. 두 버전 중 하나라도 변경 시 비교 불가 표시 |
| `prompt_version` 변경 시 과거 비교 | 과거 스냅샷은 *historical*로 표시, 변화 그래프 라인은 표시하되 **"버전 변경 구간" 점선 인디케이터** 의무. 변화 폭 비교는 같은 버전 구간 안에서만 산정 |
| `scoring_version` 변경 시 | 변화 그래프에 **수직 마커** + "산정식 업데이트" 라벨. 신구 버전 점수 직접 비교 금지 (사용자 안내 의무) |
| 메모 수정 시 점수 영향 | **0** — 메모는 점수 산정 입력에 포함되지 않음. 메모 수정은 타임라인 텍스트만 갱신, 스냅샷 점수 불변 |

> 변화 그래프 표시 정책: 같은 버전 구간 안에서는 라인, 버전 전환 구간에는 점선 + 마커. 사용자에게 *"산정식이 바뀌어 직접 비교는 의미가 다를 수 있어요"* 카피 노출.

### 4.1 메인 네비 (3탭 + 이런건어때 시트)

```
[오늘] - [합피드] - [내사주]
```

| 탭 | 핵심 컴포넌트 | LLM 호출 |
|---|---|---|
| 오늘 | 오늘합, 진행 중 진단, 최근 노트 | 오늘합(Lazy) |
| 합피드 | 인연 그리드 (**Phase 1.5+ 흐름 변화 큰 인연 자동 상단 정렬, ADR-033**), 카드 탭 → 합보기 + 타임라인 | 합보기(요청 시) |

### 4.2 핵심 사용자 흐름

**Flow A — 신규 가입자 첫 5분 (인연 등록 우선 + 2트랙 분기, ADR-032·029)**

가입 직후 입구 화면에 **설명 카피 의무 노출** (ADR-032 보강): *"먼저 떠오르는 사람을 고르면, 그 사람과의 합흐름을 보기 위해 내 생년월일도 함께 계산합니다."* — 사용자가 "내 정보도 안 넣었는데 왜 남의 생일을 먼저?"라고 느끼지 않도록 **순서 의도와 본인 정보 사용 목적을 명시적으로 안내**.

**입력 6단계 순서 (ADR-032 보강)**:
1. **오늘 떠오르는 사람 선택** — "+ 인연 등록" 진입
2. **별명 입력** (실명 금지 카피 한 줄)
3. **"생일을 아나요?" 예 / 아니오 분기**
4. **분기**:
   - **4-A (예)**: 상대 생년월일 + 시 입력 → Track A
   - **4-B (아니오)**: §3.4 *"첫만남 플레이"*(자기 성찰형) Track B
5. **내 생년월일·시 입력** (짧은 인라인 단계, 첫 진입 시만)
6. **결과 생성** → 합카드 [1][2][4] 첫 화면 즉시 인지

이후:
- Google OAuth는 1단계 *직전*에 1회 처리. 1회 무료 해석 + D+1 무료 다시합 진입 토큰 자동 지급
- (Track A) 합카드 첫 화면 즉시 인지 → 스크롤 후 [6] 푸터에서 D+1 다시합 알림 예약 / 공유 / 메모 (2차 액션, ADR-026)
- (Track B) 자기 성찰 카드 → 첫 만남 TIP 3 + 메모 → "인연을 알게 되면 정확 해석 받기" CTA로 Track A 유도
- 두 번째 해석부터 8p 안내 / 충전 10·55·120부적 CTA

> **순서 정렬 의도 (ADR-032)**: 입구가 *본인 사주*가 아니라 *오늘 떠오르는 사람*이어야 "사주보다 사람 전면" 정체성이 첫 진입에서 즉시 인지된다. 단 **사용자가 본인 정보 입력 이유를 의심하지 않도록 입구 화면에 설명 카피 의무 노출** + 별명·상대 생일 입력 후 "생일 known/unknown" 분기로 사용자 부담 분산.

**Flow B — 일상 리텐션**
1. 매일 아침 푸시 → 오늘의 합카드
2. 어제 해석 인연 D+1 알림 → "[4p로 다시 보기]" CTA
3. 매주 일요일 → 이번주 합흐름 추천

**Flow C — 결정 지원**
1. 이런건어때 시트 → 합보기 + 상황 컨텍스트 selector 진입
2. 양쪽 사주 결합 해석 → 합카드 + 상황 맞춤 일단이거해봐 3
3. 합메모 → 타임라인 기록

**Flow D — 관계 진화 회고 (Phase 1.5)**
1. D+7 푸시 → 인연 디테일 → 타임라인 뷰
2. 4p 다시합 → 합흐름 그래프
3. 메모 추가 → D+30 알림 예약

  - 95~98% → Tier 1: 어댑터 패치
  - 85~95% → Tier 2: manseryeok-js 교체 (어댑터만 교체)
  - <85% → Tier 3: 재설계 [TBD-2: 상용 만세력 API · sajupy on Cloud Run · manseryeok-js bypass 중 선택, G0 진입 전 확정]

### 6.3 데이터 모델

- **BirthProfile**: profile_id, user_id, name, birth_date, birth_time(또는 unknown), birth_place, gender, calendar(solar/lunar), is_dst_aware. 본인은 실명, 인연은 별명. 인연 별명도 LLM 페이로드 제외
- **ChartCore**: pillars(year/month/day/hour stem·branch), day_master, five_elements_balance, sinsal_tags, chart_hash
- **TheoryProfile**: profile_version, calendar_basis, time_zone_handling, sinsal_set, calculation_options
- **KnowledgeCitation**: classic_id, chapter, original_text, modern_translation, review_status

### 6.4 Privacy by design

- LLM 페이로드 = `chart_core + question_slot + theory_profile.profile_version`만
- 원시 birth_date·name·email·birth_place·별명 **절대 LLM 미전달**
- G2 게이트: 페이로드 직렬화 후 YYYY-MM-DD·이메일 정규식 grep 테스트, 검출 시 빌드 실패
- **공유합카드 표시 범위 사용자 선택**: 매 공유 시 *별명만 / 별명+오행 / 별명+성별* 명시 선택. 기본값 = 별명만 + 합점수. 생일·시각·출생지는 어떤 옵션에서도 비노출
- **출생시간 미상 시나리오 추정 (UX 강화)**: 인연 시간 미상 시 12지 기반 시나리오 추정 + 신뢰 구간(±) 표시. 합카드에 **"시나리오 추정 모드" 배지를 헤더 [1] 옆 고정 위치 + 컬러 그라디언트 강조 + 합점수 [2] 옆 신뢰 구간(±) 인라인 표기**로 의무 노출. 단정 톤 회피 카피("정확히/확실히") 자동 필터 강화

| 오늘의 합카드 캐시 | `user_id + date` → `daily_cards` | 당일만 |

캐시 키:
```
sha256(chart_hash + question_slot + prompt_version + theory_profile.profile_version)
```

### 7.3 프롬프트 주입 방어

- 입력 sanitize: HTML/템플릿 기호 제거, 주입 키워드 필터, 200자 제한
- 프롬프트 구조 고정: 유저 입력은 `<user_input>...</user_input>` 안쪽
- 출력 후처리: `banned_phrases_catalog.yaml` 필터, JSON 스키마 검증, 시스템 프롬프트 노출 패턴 차단
- Rate limit: IP 기반 분당 10 / 시간당 60

### 7.4 API 키

- 저장: Supabase Edge Function Secrets (클라이언트 번들 절대 금지)
- 분리: Production / Staging / CI-Regression 독립 키
- 로테이션: 분기 1회 정기 + 누출 의심 즉시
- 런북: `docs/runbooks/api-key-rotation.md`

### 7.5 ZDR·로그

- OpenAI production project는 ZDR(Zero Data Retention) 적용 확인이 필수이며 `OPENAI_PROJECT_ID`로 명시 routing한다.
- OpenAI API key와 `OPENAI_PROJECT_ID`는 같은 organization/project 범위에서 발급·연결한다.
- Anthropic fallback은 외부 LLM sub-processor로 취급한다. Production fallback key 설정 전 보관·학습 미사용·DPA/계약 상태를 확인하고, LLM payload는 PII 5필드 + gender 원본을 제외한 `chart_core + question_slot + theory_profile.profile_version` 범위로 제한한다.
- Privacy Policy에는 OpenAI core와 Anthropic fallback을 LLM sub-processor로 명시하되, OpenAI는 ZDR 적용 project로 운영한다.

### 7.6 프롬프트 카나리

- `prompt_versions` 테이블 (active/canary/rolled_back + canary_ratio)
- 신버전: 5% 카나리 → 72시간 모니터링 → 승격 또는 즉시 롤백
- 즉시 롤백 트리거:
  - `banned_phrases` 감지율 > 3% (5분 기준)
  - LLM-as-judge 평균 < 3.0
  - 유저 👎 비율 > 20%
- 롤백: `UPDATE prompt_versions SET status='active' WHERE version='vX'` (SQL 1줄)
- **구현 상태 (2026-05-28)**: `loadPromptForUser(client, promptName, userId)` 가 deterministic SHA-256 sampling 으로 active/canary 분기. 6모드 + today_with_relation + daily_hap 모두 DB-backed. seed-prompts.ts 의 `> CanaryVersion:` `> CanaryRatio:` frontmatter 로 동시 시드.

각 모드는 5-7문항 응답 + 양쪽 사주로 진단. *명리적 관점*은 §3.2.1 산정식의 가중치가 모드별로 ±10p 강조되는 방식 (LLM 컨텍스트 + 합카드 "근거 보기" 동일 사용).

| 모드 | 명리 강조 | 5-7문항 예시 | 합카드 |
|---|---|---|---|
| 일합 | 십신(정관·편관·식상) — 권위·실행 | 협업 빈도/커뮤니케이션/스트레스/목표 일치도 | 호환성 / 강점 / 주의점 / 협업 팁 3 |

| 첫합 | 천간합·지지합 첫 발현 + 도화·홍염 | 첫인상/대화/관심사/가치관 | 합점수 / 진척 가능성 / 다음 만남 가이드 |
