# 합점수(호환성) 점수 계산 스펙

> 본 문서는 **개발 명세**이다. ADR-035에 따라 기획서(`fluttering-gathering-island.md`)의 §3.2.1 가중치 외 *상세 점수 산정 명세*를 분리 관리한다. Phase 0 G1 단계에서 명리 specialist가 채우고 검수한다.

**버전 관리**: `scoring_version` 필드로 추적. prompt_version과 별개. 카나리 5%·즉시 롤백 가능 (ADR-008과 동일 방식). **현행 `SCORING_VERSION = 2`** (2026-06-11, §10 변경 이력 참조).

**결정성 원칙**: 동일 입력 → 동일 점수. LLM은 점수 산출에 개입 금지.

## 1. 점수 산정 구조

```
final_score = clamp(
  weight_hap_chung_hyung_hae * S_hap_chung_hyung_hae
  + weight_sipsin * S_sipsin
  + weight_ohaeng * S_ohaeng
  + mode_adjustment(mode)
  ,
  0, 100
)
```

|---|---|
| `weight_hap_chung_hyung_hae` | 0.40 |
| `weight_sipsin` | 0.30 |
| `weight_ohaeng` | 0.30 |

**모드별 ±10p 가중치 재분배**: §6 표 참조.

## 2. 합·형·충·해 점수표 (S_hap_chung_hyung_hae)

> 각 항목당 0-100 정규화. 양 일주(연주·월주·일주·시주) 4쌍에서 발생한 매핑을 합산 후 정규화.

### 2.1 천간합 (긍정)

| 합 | 점수 |
|---|---|
| 갑기합 (甲己合) | +12 |
| 을경합 (乙庚合) | +12 |
| 병신합 (丙辛合) | +12 |
| 정임합 (丁壬合) | +12 |
| 무계합 (戊癸合) | +12 |

### 2.2 지지합 (긍정)

| 합 | 점수 |
|---|---|
| 자축합 (子丑合) | +10 |
| 인해합 (寅亥合) | +10 |
| 묘술합 (卯戌合) | +10 |
| 진유합 (辰酉合) | +10 |
| 사신합 (巳申合) | +10 |
| 오미합 (午未合) | +10 |

### 2.3 삼합·반합 (긍정)

| 삼합 | 반합 |
|---|---|
| 인오술 화국 (寅午戌 火局) +15 | 인오·오술 +8 |
| 신자진 수국 (申子辰 水局) +15 | 신자·자진 +8 |
| 사유축 금국 (巳酉丑 金局) +15 | 사유·유축 +8 |
| 해묘미 목국 (亥卯未 木局) +15 | 해묘·묘미 +8 |

### 2.4 형·충·파·해 (부정)

| 항목 | 점수 |
|---|---|
| 충 (沖, 6충) | −15 |
| 형 (刑, 삼형·자형) | −10 |
| 파 (破) | −5 |
| 해 (害) | −5 |

### 2.5 합·충 동시 발생 우선순위·중첩 처리

- **합과 충이 동일 일주에 동시 발생 시**: 합·충 *각각 산정 후 합산*. 단 **합 우선 가중 1.2x** 적용 (관계 회복력 반영)
- **삼합 + 충 동시**: 삼합 +15 그대로, 충 −15 → 0이 아닌 +0 (양수 우세)
- **중복 합** (예: 일주에 천간합·지지합 동시): 합산 시 +5 보너스
- **중복 충** (다층 충): 합산 시 −3 페널티

### 2.6 정규화

```
S_hap_chung_hyung_hae = clamp(
  ((sum_score + 50) / 100) * 100  // -50~+50 범위를 0-100으로 매핑
  ,
  0, 100
)
```

**예시**: 합 +30, 충 −10 → sum_score = +20 → S = 70

## 3. 십신 상호작용 (S_sipsin)

본인 일간 기준 인연의 십신을 산정. 십신 = 비견·겁재·식신·상관·정재·편재·정관·편관·정인·편인.

### 3.1 모드별 *권위·실행·정서·자산* 축 매핑

| 십신 | 권위 (직장) | 실행 (비즈니스) | 정서 (친구·연애) | 자산 (재물) |
|---|---|---|---|---|
| 정관 | +12 | +5 | +3 | +5 |
| 편관 | +8 (스트레스 +) | +10 | −3 | +3 |
| 식신 | +5 | +8 | +10 | +5 |
| 상관 | +3 (마찰) | +10 (창의) | +5 | +3 |
| 정재 | +5 | +8 | +5 | +12 |
| 편재 | +3 | +10 | +3 | +10 |
| 정인 | +8 | +5 | +10 | +5 |
| 편인 | +5 | +5 | +5 | +5 |
| 비견 | +3 (경쟁) | +5 | +8 | +3 |
| 겁재 | −3 (경쟁) | +3 | +3 | −3 |

### 3.2 정규화

```
S_sipsin = clamp(
  ((mode_axis_score + 30) / 60) * 100
  ,
  0, 100
)
```

## 4. 오행 균형 (S_ohaeng)

양 사주의 오행 5지표(목·화·토·금·수)를 0-100% 비율로 산출 후 *상호 보완·과다 중첩*을 측정.

### 4.1 보완 보너스 / 과다 중첩 페널티

```
for element in [목, 화, 토, 금, 수]:
  본인_부족 = max(0, 20 - 본인_비율)   // 20% 미만이면 부족
  인연_보완 = min(인연_비율, 본인_부족)
  보완_보너스 += 인연_보완 * 1.5

  본인_과다 = max(0, 본인_비율 - 30)   // 30% 초과면 과다
  인연_중첩 = min(인연_비율, 본인_과다)
  중첩_페널티 += 인연_중첩 * 1.0
```

### 4.2 정규화

```
S_ohaeng = clamp(
  50 + (보완_보너스 - 중첩_페널티)
  ,
  0, 100
)
```

기본 50점에서 보완 ↑ / 중첩 ↓.

## 5. 시간 미상 시 시나리오 추정 (ADR-024 결합)

인연 시간 미상 시 12지(子·丑·寅·...·亥) 시나리오 추정.

### 5.1 산정 절차

1. 12지 각각에 대해 위 1~4 산정식으로 점수 12개 산출
2. *최소·최대*: `min_score`, `max_score`
3. 표시 점수 = 평균: `mean(12 scores)`
4. **신뢰 구간 ±**: `(max_score - min_score) / 2`

### 5.2 표시 규칙

```
display_score = round(mean(scores))
display_range = ±round((max - min) / 2)
```

예: 12지 점수 분포 60-78 → 표시: **69 ± 9**

신뢰 구간이 **±15 이상**이면 "시나리오 추정 모드" 배지 + *"시간 정보가 추가되면 더 정확해져요"* CTA 의무 (ADR-024).

## 7.5 오늘온도 표시 변환 (2026-05-21 사용자 승인)

`compat_score`는 DB/API/스냅샷/정렬의 canonical 값으로 계속 0-100 결정형 점수를 유지한다. 사용자에게 노출되는 대표 지표만 `오늘온도`로 변환한다.

```
today_temperature = clamp(
  37.0 + ((compat_score - 50) / 20),
  35.5,
  38.9
)
display_temperature = round(today_temperature, 1) + "°C"
```

표시 기준:

| 내부 점수 | 표시 |
|---|---|
| 50 | 37.0°C |
| 60 | 37.5°C |
| 62 | 37.6°C |
| 40 | 36.5°C |
| 35 | 36.3°C |

- 분석한 날의 관계 흐름이 좋으면 `37.5°C`를 초과한다.
- 분석한 날의 관계 흐름이 나쁘면 `36.5°C` 미만으로 표시한다.
- 변화량(`change_score`, `delta_vs_yesterday`)은 동일 기울기(`20점 = 1.0°C`)로 변환해 `+0.8°C`처럼 표시한다.
- LLM은 오늘온도 산출에도 개입하지 않는다. 오늘온도는 `compat_score`의 결정형 표시 변환이다.

## 8. 결정성 보장

### 8.1 캐시 키
```
sha256(
  본인_chart_hash + 인연_chart_hash + mode + scoring_version + 일진_date
)
```

### 8.2 검증 절차 (Phase 0 G1)

1. 동일 입력 1,000회 호출 → 동일 점수 반환 100%
2. 다른 시간대(KST/UTC) 호출 → 일진 date 정규화 후 동일 점수
3. scoring_version 변경 후 동일 입력 → 점수 변화 허용, 단 변화 그래프에 마커 의무

### 8.3 LLM 격리

LLM은 본문 작성 시 *점수 입력값으로만 받음*. 점수 산출 자체에 절대 개입 금지. 점수 관련 LLM 출력 시 grep 테스트로 차단.

### 8.4 파생·교차분석층 무접촉 (ADR-040, 2026-06-11)

`chart_core.derived`(SajuDerived)와 `cross_analysis`(CrossAnalysis)는 **LLM 해석 근거
전용**이며 본 spec의 어떤 점수 컴포넌트에도 입력되지 않는다. 알고리즘·검수 체크리스트는
`docs/specs/manseryeok_theory.md` §6 참조. 해당 층의 변경은 SCORING_VERSION을 움직이지
않는다 (점수식 무접촉 — 같은 라운드의 SCORING_VERSION 2는 C1 yunse 인코딩 버그픽스로 별건).

## 8.5 yunse_adjustment (Phase Y3 — 사용자 §1.1 확정 2026-05-07)

yunse 4-layer를 결정형 가중치로 합점수에 반영. LLM 산출 금지 (ADR-035).

### 산식

```
yunse_adjustment = clamp(
  10 × (0.40×Δ_daeun + 0.30×Δ_seyun + 0.20×Δ_wolun + 0.10×Δ_iliun) × MODE_FACTOR,
  -10, +10
)
```

### Δ 함수 (단순화 — 사용자 승인)

```
Δ(layerPillar, relDayPillar) = clamp((천간합?+1:0) + (지지합?+1:0) + (지지충?-1:0), -1, +1)
```

- **판정 방식 (SCORING_VERSION 2, 2026-06-11)**: 양쪽 기둥을 한자로 정규화(한글 입력은 변환, 한자는 통과)한 뒤 §2.1 천간합 / §2.2 지지합 / §2.4 충 한자 테이블 lookup
  - v1은 한글 배열 인덱스 매칭(갑-기 간격 5 등)이라 프로덕션 ganji(한자 — ssaju yunse·KASI day_pillar)에서 검출이 항상 실패해 `yunse_adjustment`가 0으로 고정되는 버그가 있었다. v2는 인코딩 매칭만 수정 — 점수 의미(+1/+1/−1, clamp ±1)는 v1과 동일
- 형·파·해·삼합·반합 미포함 (단순화)

### MODE_FACTOR

일합 1.00 / 첫합 0.95 / 썸합 0.90 / 돈합 0.85 / 친구합 0.80 / 오래합 0.70

### 구현체

`src/lib/scoring/yunseAdjustment.ts`
